# МОДУЛЬ 1: Основы промпт-инжиниринга
Промпт-инжиниринг — это дисциплина проектирования входных данных для LLM с целью получения точных, воспроизводимых и безопасных ответов.
Ключевые компоненты промпта:
- Role (Роль): задаёт экспертный контекст (напр. "Ты senior Python-разработчик и специалист по безопасности кода").
- Context (Контекст): фоновые данные, ограничения, целевая аудитория.
- Instruction (Инструкция): чёткое действие, которое должна выполнить модель.
- Format (Формат вывода): JSON, Markdown, таблица, список шагов.
- Constraints (Ограничения): "не выдумывай", "используй только указанные библиотеки", "объём до 300 слов".

# МОДУЛЬ 2: Техники повышения качества
- Zero-shot: промпт без примеров. Работает для простых задач.
- Few-shot: добавление 2-5 примеров "вход → ожидаемый выход". Резко снижает галлюцинации.
- Chain of Thought (CoT): инструкция "Думай пошагово". Активирует внутренние механизмы рассуждения модели.
- Self-Consistency: генерация нескольких CoT-вариантов и выбор наиболее частого/согласованного ответа.
- ReAct (Reason + Act): чередование мыслей, действий и наблюдений. Идеально для агентов с доступом к инструментам.

# МОДУЛЬ 3: RAG и работа с контекстом
LLM имеет ограниченное окно контекста. Прямая передача всей документации приводит к потере внимания (attention dilution).
RAG (Retrieval-Augmented Generation) решает это через:
1. Индексацию: разбиение документов на чанки, эмбеддинг, сохранение в векторной БД.
2. Retrieval: поиск top-k наиболее релевантных чанков по запросу (MIPS / ANN).
3. Augmentation: склейка запроса + найденных чанков в финальный промпт.
Best Practices:
- Чанки: 300-500 токенов с перекрытием (overlap) 10-20%.
- Метаданные: добавляйте источник, версию, теги в каждый чанк.
- Hybrid Search: комбинация векторного (семантика) и BM25 (точные ключевые слова).

# МОДУЛЬ 4: Интеграция с IDE и агентами
Cursor, GitHub Copilot, Devin и другие инструменты используют LLM как "мозг" агента.
- Указывайте область контекста явно: @workspace, @files, @symbols.
- Используйте .cursorrules для глобальных ограничений проекта.
- Проверяйте diff перед принятием изменений (LLM может вносить синтаксические или логические ошибки).
- Итерация важнее идеального первого промпта. Используйте self-reflection: "Проверь код на edge-cases, исправь, если нужно".

# МОДУЛЬ 5: Оценка и отладка промптов
Метрики: точность, полнота, время генерации, токсичность/галлюцинации.
Инструменты: LangSmith, Arize Phoenix, OpenEval, ручные A/B тесты.
Цикл улучшения: Запрос → Ответ → Оценка → Уточнение промпта/контекста → Повтор.
